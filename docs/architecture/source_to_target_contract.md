# Source-to-Target Contract

## Purpose

This document defines how source data from France Travail API is mapped into the Job Market Radar analytical model.

The contract helps ensure that ingestion, dbt transformations, validation checks, and documentation all agree on the same expected data flow.

---

## MVP Source

### Source Name

```text
france_travail
```

### Source Type

Public job API.

### Source Purpose

Collect job postings related to Data Engineering, ETL, analytics engineering, and adjacent junior-friendly data roles in France and Europe.

---

## Target Architecture

```text
France Travail API
  -> raw.raw_france_travail_job_postings
  -> staging.stg_france_travail_job_postings
  -> warehouse.wh_job_posting_snapshots
  -> warehouse.wh_job_posting_current
  -> marts.*
```

---

## Key Identifiers

| Identifier | Layer | Description |
|---|---|---|
| `batch_id` | raw and downstream | One pipeline execution |
| `request_id` | raw and staging | One API request/page/search parameter set |
| `raw_job_posting_id` | raw and downstream lineage | One raw job posting row |
| `source_job_id` | raw and downstream | Job id from the source API |
| `source_job_key` | raw and downstream | Stable key: `france_travail:{source_job_id}` |
| `canonical_job_key` | warehouse and marts | Canonical job key; equals `source_job_key` in MVP |
| `search_scope_key` | raw and downstream | Comparable search configuration |

## Raw Contract

### Target Table

```text

```

### Grain

| `batch_id` | Created by pipeline | Yes | Links to `raw.raw_load_batches` |

| `payload_hash` | Hash of raw payload | Yes | Used to detect changes |
| `ingested_at` | Pipeline timestamp | Yes | Technical ingestion time |

## Current-State Contract

### Target Model

```text
warehouse.wh_job_posting_current
```

### Grain

One row per latest known canonical job posting.

### Logic

The current-state model is derived from snapshots.

For each `canonical_job_key`:

- determine `first_seen_at`
- determine `last_seen_at`
- keep latest known attributes
- determine whether the job is active in the latest comparable search scope
- preserve latest `payload_hash`

### Important Constraint

A job can only be marked inactive if it is missing from a later batch with the same `search_scope_key`.

| `skill_name` | Display skill name |
| `match_source` | Example: `dictionary_rule` |

### Inputs

- `warehouse.wh_skill_dictionary_versions`

### Required Output

| Field | Description |
|---|---|
| `relevance_score` | Numeric score from 0 to 100 |
| `relevance_reason` | Human-readable explanation |
| `matched_candidate_skills` | Candidate skills found in the job |
| `missing_growth_skills` | Useful skills missing from candidate profile |
| `is_junior_friendly` | Whether the job appears suitable for junior/career-switcher profile |

## Validation Contract Summary

Raw validation:

- `batch_id` is not null
- `request_id` is not null
- `source_job_id` is not null
- `raw_payload_json` is not null
- `payload_hash` is not null

Staging validation:

- `source_job_key` is not null
- `job_title` is not null
- dates are valid where available
- controlled values are respected where possible

Warehouse validation:

- `canonical_job_key` is not null
- one current row per `canonical_job_key`
- snapshots link back to raw rows
- `first_seen_at <= last_seen_at`

Marts validation:

- `relevance_score` is between 0 and 100
- active job counts are non-negative
- marts are not empty after successful ingestion
- weekly summary grain is respected

---

## Contract Limitations

The MVP contract intentionally avoids:

- advanced cross-source deduplication
- machine learning relevance models
- complex location geocoding
- real-time job status verification

These limitations should be documented clearly in the README and architecture docs.
